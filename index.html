  • The CERN directorate released the source code of WorldWideWeb into the public domain.
  • January 1995
  • The first web accessibility guideline was compiled by Gregg Vanderheiden.
  • 1995 => 1998
  • Over the next few years, more than 38 different Web access guidelines were compiled by various authors and organizations.
  • 1997
  • The W3C launched the Web Accessibility Initiative (WAI) with endorsement by The White House and W3C members.
  • 1998
  • These were brought together in the "Unified Web Site Accessibility Guidelines" compiled at the University of Wisconsin-Madison. Version 8 of the it, served as the starting point for the W3C's WCAG 1.0

Main Principles For Accessible Web

Perceivable

  • 1.1: Provide text alternatives for any non-text content so that it can be changed into other forms people need, such as large print, braille, speech, symbols or simpler language.
  • 1.2: Time-based media: Provide alternatives for time-based media.
  • 1.3: Create content that can be presented in different ways (for example simpler layout) without losing information or structure.
  • 1.4: Make it easier for users to see and hear content including separating foreground from background.

Operable

  • 2.1: Make all functionality available from a keyboard.
  • 2.2: Provide users enough time to read and use content.
  • 2.3: Do not design content in a way that is known to cause seizures.
  • 2.4: Provide ways to help users navigate, find content, and determine where they are.

Understandable

  • 3.1: Make text content readable and understandable.
  • 3.2: Make web pages appear and operate in predictable ways.
  • 3.3: Help users avoid and correct mistakes.

Robust

  • 4.1.: Maximize compatibility with current and future user agents, including assistive technologies.

Core concepts

Semantic DOM

Use the correct DOM elements where nedded (for example do not use <a> as an alternative to <button>). Use semantic DOM architecture devisions (e.g. <nav>, <section>, <article> etc.)

By using semantic tags and styles instead of using presentational tags, which are only visual, the meaning is not carried through by a technology that only reads the code.

Tabbable Content

This is the most common way to navigate in a web page with assistive technologies (and regular keyboards). For some people it's the only way to navigate in a web page.

Use tabindex, block skip links and, tab trappings when necessary (we will talk about how on a saparate talk).

Visual Focus

Often we tend to hide this feature. It helps keyboard users (but not only), to orientate in the page, and to understand which elements are interactive.

We should keep in mind that this feature (and a few more), was implemented with a good reason, and not to be hasty when deciding to override it without an alternative.

Color Contrast

When someone has low visual acuity, certain color palettes on websites may make it difficult for that person to fully see and understand the website content.

Low-quality displays, displays with low backlight, and low-contrast display settings can all reduce the default amount of contrast, making already-low-contrast designs even lower.

level AA requires a contrast ratio of 4.5:1 for normal text and 3:1 for large text.

Alternate Text

Alternative text provides a textual alternative to non-text content in web pages.

Adding alternative text for images is the first principle of web accessibility. It is also one of the most difficult to properly implement. The web is replete with images that have missing, incorrect, or poor alternative text.

ARIA attributes

WAI-ARIA if you want to be formal, is a set of attributes that you can add to HTML elements.

These attributes communicate role, state and property semantics to assistive technologies via the accessibility APIs implemented in browsers.
